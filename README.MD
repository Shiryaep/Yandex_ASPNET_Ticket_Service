## Настройка базы данных

Приложение использует PostgreSQL в качестве основной базы данных. Строка подключения настраивается в файле `appsettings.json` (или `appsettings.Development.json` для среды разработки).

Пример строки подключения:

```json
"ConnectionStrings": {
  "DefaultConnection": "Host=localhost;Port=5432;Database=TicketServiceDb;Username=postgres;Password=your_password"
}
```
**ВНИМАНИЕ:** Для каждого сервиса порт базы данных будет отличаться. Смотрите конктретные порты в `README` для каждого сервиса!

Замените `your_password` на пароль пользователя PostgreSQL. Убедитесь, что база данных `TicketServiceDb` существует, либо оставьте пустой — приложение автоматически создаст её при первом запуске (см. ниже).

### Управление схемой через миграции EF Core

Схема базы данных управляется с помощью миграций Entity Framework Core. Это позволяет безопасно применять изменения структуры базы данных, сохраняя историю миграций и обеспечивая согласованность между различными окружениями.

#### Создание новой миграции

После изменения моделей или конфигураций Entity Framework выполните команду для создания миграции:

```bash
dotnet ef migrations add <MigrationName> --project Infrastructure --startup-project Presentation --output-dir Migrations
```

- Замените `<MigrationName>` на описательное имя миграции (например, `AddNewColumnToEvents`). Миграция будет создана в директории `Migrations` проекта `Infrastructure`.
- Не меняйте аргументы `--project` и `--startup-project`
- Запускайте команду в папке проекта, для которого нужно добавить миграцию. **НЕ запускайте команду в корне проекта**!

#### Применение миграций к базе данных

Чтобы применить все `pending` миграции к базе данных, выполните:

```bash
dotnet ef database update --project Infrastructure --startup-project Presentation
```

Эта команда обновит схему базы данных до последней миграции. При первом запуске она создаст все таблицы, определённые в миграциях.

#### Откат миграции

Если необходимо откатить последнюю применённую миграцию, используйте:

```bash
dotnet ef database update <PreviousMigrationName> --project Infrastructure --startup-project Presentation
```

Для удаления миграции после ее отката, используйте:

```bash
dotnet ef migrations remove --project Infrastructure --startup-project Presentation
```

> **Примечание:** Убедитесь, что у вас установлен глобальный инструмент EF Core: `dotnet tool install --global dotnet-ef`. Если инструмент не установлен, выполните установку перед использованием команд.

<a name="features" id="features"></a>
## Особенности приложения
### Архитектурные особенности

1. Приложение с этого момента разделено на несколько независимых сервисов, между которыми отсутствует синхронное взаимодействие. 

 - Сервис `YaUsers` отвечает за регистрацию пользователей и выдачу токенов
 - Сервис `YaEvents` отвечает за управление событиями - их получение и создание
 - Сервис `YaBookings` отвечает за создание и отклонение бронирований

 2. Все сервисы запускаются как отдельные приложения:

 - Сервис `YaUsers` использует порт `7013`
 - Сервис `YaEvents` использует порт `7011`
 - Сервис `YaBookings` использует порт `7012`

3. Каждый сервис разделен на четыре слоя в соответствии с Clean Architecture

- Domain — доменные сущности, value objects, доменные исключения. Не зависит ни от чего внешнего.
- Application — use cases, сервисы, интерфейсы портов (репозитории, шлюзы), DTO. Зависит только от Domain.
- Infrastructure — реализации портов: репозитории, DbContext, внешние клиенты. Зависит от Application и Domain.
- Presentation — контроллеры/Minimal API эндпоинты, HTTP-маппинг, регистрация зависимостей. Зависит от Application и Infrastructure.

4. Регистрация зависимостей происходит через вызов extension методов, что позволяет сделать Program.cs компактнее.

## Формат ошибок

API использует стандартный формат ошибок [Problem Details (RFC 7807)](https://datatracker.ietf.org/doc/html/rfc7807). В случае ошибки клиент получит ответ с соответствующим HTTP‑кодом и JSON‑объектом, содержащим поля `type`, `title`, `status`, `detail` и, при необходимости, `instance`.

Пример ответа с ошибкой 404:

```json
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.4",
  "title": "Not Found",
  "status": 404,
  "detail": "Event with id [12345678-1234-1234-1234-123456789abc] not found",
  "instance": "/events/12345678-1234-1234-1234-123456789abc"
}
```

<a name="auth" id="auth"></a>
## Безопасность и права

В проекте реализован эндпоинт для регистрации и входа пользователя. На данный момент есть две роли - User и Admin. В соответствии с этим разграничением некоторые узлы API доступны только определенным категориям пользователей:

**Доступно всем**
```http
   GET /api/events — получить список событий
   GET /api/events/{id} — получить событие по ID

   POST /api/Auth/register - зарегистрироваться в приложении
   POST /api/Auth/login - войти в приложение
```

**Доступно пользователям**
```http
   POST /api/events/{id}/book — создать бронирование для указанного события
   GET /api/bookings/{id} — получить информацию о бронировании по его ID
   DELETE /api/bookings/{id} - отменить бронирование по его ID
```

**Доступно только администраторам**
```http
   POST /api/events — создать новое событие
   PUT /api/events/{id} - обновить событие по ID
   DELETE /api/events/{id} - удалить событие по ID
```

### При несанкционированном доступе
- При попытке доступа к любым API узлам без авторизации в категориях "Доступно пользователям" и "Доступно только администраторам" будет возвращена ошибка 401 - Unauthorized
- При попытке доступа к админским узлам с правами пользователя будет возвращена ошибка 403 - Forbidden

<a name="jwt" id="jwt"></a>
## Получение токена и настройка JWT

В проекте используется JWT для получения и проверки токена. 

### Получение токена в Swagger

1. Запустите приложение
2. Отправьте запрос на `/api/Auth/Register` со своими данными - так вы зарегистрируетесь в приложении. Сделать это надо единожды - во второй раз будет возвращена ошибка 409 - Conflict, так как пользователь будет уже добавлен.
3. Отправьте запрос на `/api/Auth/Login` со своими данными - в ответ вы получите действующий токен в виде длинной строки с символами.
4. Нажмите на кнопку `Authorize` и введите `Bearer ВАШ-СКОПИРОВАННЫЙ-ТОКЕН` и нажмите `Authorize` - вы получите доступ к API со своими правами

### Настройка JWT

Для настройки JWT в приложении вам необходимо заменить значения в файле `appsettings.json`, а именно:
- В секции Jwt
```
    Key - ваш ключ безопасности для подписи токена. 
    Issuer - сервис, который выдает токены.
    Audience - сервси, для которого выдается токен.
    Lifetime - время в минутах, в течение которого токен актуален.
```
**ВНИМАНИЕ** 
- В Production окружении необходимо использовать длинный и сложный ключ!
- Нельзя коммитить Production ключ, т.е. нельзя комитить изменения в `Jwt` секцию файла `appsettings.json`.
- Держите настоящий файл в безопасном месте, не передавайте его по сети. В репозитории файл присутствует только для примера.

## Docker
Для запуска приложения вам потребуется докер - о нем уже много раз упоминалось выше. Вам нужно собрать и запустить контейнеры.

На данный момент в докере будет запущено 9 контейнеров:
- 3 контейнера с базами данных, чтобы сервисы могли подключиться к ним на старте, иначе вы получите ошибку "Невозможно подключиться к серверу"
- 1 контейнер с UI приложением к базе данных, для удобства управления ею
- 3 контейнера с микросервисами
- 1 контейнер с Kafka для общения сервисов между друг другом
- 1 контейнер с UI для Kafka

### Контейнеры с приложениями

Три микросервиса теперь собираются в образ через собственные файлы `Dockerfile`, которые можно найти в папках сервисов. Вам не требуется собирать образы отдельно, так как все манипуляции прописаны в `docker-compose.yml` (см. далее)

### Docker Compose
1. Скопируйте файл `.envexample` в корень проекта и переименуйте его в просто `.env`.
2. Замените в файле все строки начинающиеся с `___` на ваши значения. Например `POSTGRES_DB=___Name of your db like "mydatabase"` замените на `POSTGRES_DB=mydatabase`. Не берите пароли и названия в кавычки!
3. В корне проекта откройте консоль и выполните `docker-compose up -d`. Контейнер автоматически соберется, скачает нужные образы, подцепит переменные из `.env` файла и запустит все необходимые контейнеры.
4. Можно подключаться к сервисам через Swagger (см. README сервисов)

> **Совет:** Если вы уже поднимали контейнеры, то у вас, скорее всего, остались файлы от предыдущей конфигурации. Если вы будете менять конфигурацию, то используйте команду `docker compose down -v --remove-orphans; docker compose up -d`, чтобы сначала очистить все хвосты, а потом уже поднимать новые контейнеры.

<a name="kafka" id="kafka"></a>
## Kafka

В проекте используется Kafka для взаимодействия между сервисами:

1. При запуске `HostedService` в `YaEvents` проверяет топик `booking-confirmed` на существование. Если топика нет, то сервис его создает.
2. Сервис `BookingBackgroundService` в `YaBookings` публикует событие `BookingConfirmed` в топик `booking-confirmed` после того, как находит `Pending` бронирование и обрабатывает его.
3. Сервис `BookingConfirmedSubscriber` в `YaEvents` получает событие `BookingConfirmed` и обрабатывает его, уменьшая количество доступных мест.
