# ASP.NET Core Web API

Это веб‑API на базе ASP.NET Core для бронирования билетов на мероприятие.

## Требования

Перед запуском убедитесь, что у вас установлены:

- [.NET SDK 10.0 или выше](https://dotnet.microsoft.com/download)
- [Visual Studio 2026](https://visualstudio.microsoft.com/) или [Visual Studio Code](https://code.visualstudio.com/)
- [PostgreSQL 15 или выше](https://www.postgresql.org/download/)

## Настройка базы данных

Приложение использует PostgreSQL в качестве основной базы данных. Строка подключения настраивается в файле `appsettings.json` (или `appsettings.Development.json` для среды разработки).

Пример строки подключения:

```json
"ConnectionStrings": {
  "DefaultConnection": "Host=localhost;Port=5432;Database=TicketServiceDb;Username=postgres;Password=your_password"
}
```

Замените `your_password` на пароль пользователя PostgreSQL. Убедитесь, что база данных `TicketServiceDb` существует, либо оставьте пустой — приложение автоматически создаст её при первом запуске (см. ниже).

### Управление схемой через миграции EF Core

Схема базы данных управляется с помощью миграций Entity Framework Core. Это позволяет безопасно применять изменения структуры базы данных, сохраняя историю миграций и обеспечивая согласованность между различными окружениями.

#### Создание новой миграции

После изменения моделей или конфигураций Entity Framework выполните команду для создания миграции:

```bash
cd .\Yandex_ASPNET_Ticket_Service\
dotnet ef migrations add <MigrationName> --output-dir Migrations
```

Замените `<MigrationName>` на описательное имя миграции (например, `AddNewColumnToEvents`). Миграция будет создана в папке `Migrations` проекта.

#### Применение миграций к базе данных

Чтобы применить все pending миграции к базе данных, выполните:

```bash
cd .\Yandex_ASPNET_Ticket_Service\
dotnet ef database update
```

Эта команда обновит схему базы данных до последней миграции. При первом запуске она создаст все таблицы, определённые в миграциях.

#### Откат миграции

Если необходимо откатить последнюю применённую миграцию, используйте:

```bash
dotnet ef database update <PreviousMigrationName>
```

Или для полного удаления всех таблиц (осторожно!):

```bash
dotnet ef database drop --force
```

> **Примечание:** Убедитесь, что у вас установлен глобальный инструмент EF Core: `dotnet tool install --global dotnet-ef`. Если инструмент не установлен, выполните установку перед использованием команд.

## Проверка работоспособности

После запуска откройте в браузере:

```bash
https://localhost:7011/swagger
```

Вы должны увидеть интерфейс Swagger UI со списком доступных API‑эндпоинтов.

### Доступные эндпоинты

#### Управление событиями (Events)

    GET /api/events — получить список событий

    GET /api/events/{id} — получить событие по ID

    POST /api/events — создать новое событие

    PUT /api/events/{id} - обновить событие по ID

    DELETE /api/events/{id} - удалить событие по ID

#### Бронирование билетов (Bookings)

    POST /api/events/{id}/book — создать бронирование для указанного события

    GET /api/bookings/{id} — получить информацию о бронировании по его ID

### Коды ответа бронирования

Эндпоинт `POST /api/events/{id}/book` может возвращать следующие HTTP‑коды:

- `202 Accepted` – бронирование успешно создано (статус `Pending`).
- `409 Conflict` – нет свободных мест (выбрасывается исключение `NoAvailableSeatsException`).
- `400 Bad Request` – неверные параметры запроса.
- `404 Not Found` – событие с указанным `id` не существует.

## Фильтрация и пагинация событий

Эндпоинт `GET /api/events` поддерживает следующие параметры запроса:

```
[String]    title - фильтр по заголовку 
[DateTime]  from  - фильтр по дате начала (события, начинающиеся с этой даты или позже)
[DateTime]  to    - фильтр по дате окончания (события, заканчивающиеся до этой даты или раньше)
[Int]       page  - номер страницы (по умолчанию: 1)
[Int]       pageSize - количество событий на странице (по умолчанию: 10)
```

Ответ содержит список событий в формате:

```json
{
  "items": [ ... ],
  "totalCount": 13,
  "page": 1,
  "pageSize": 10
}
```

## Модель Booking и статусы

Бронирование представлено сущностью `Booking` со следующими полями:

```
[Guid]           Id - Уникальный идентификатор бронирования
[Guid]           EventId - Идентификатор события, для которого создано бронирование
[BookingStatus]  Status - Текущий статус бронирования
[DateTime]       CreatedAt - Дата и время создания бронирования
[DateTime?]      ProcessedAt - Дата и время обработки бронирования, заполняется при смене статуса
```

### Статусы бронирования (`BookingStatus`)

- **Pending** — бронирование создано и ожидает обработки.
- **Confirmed** — бронирование успешно подтверждено (билет забронирован).
- **Rejected** — бронирование отклонено (например, из‑за отсутствия свободных мест).

При создании бронирование получает статус `Pending`. Фоновая служба `BookingResolverService` периодически переводит его в статус `Confirmed`.

## Модель Event

Событие представлено сущностью `Event` со следующими полями:

```
[Guid]      Id - Уникальный идентификатор события
[String]    Title - Название события
[String?]   Description - Описание
[DateTime]  StartAt - Дата и время начала
[DateTime?] EndAt - Дата и время окончания
[Int]       TotalSeats - Общее количество мест на событие
[Int?]      AvailableSeats - Количество свободных мест
```

Поля `TotalSeats` и `AvailableSeats` используются для управления доступностью мест:
- `TotalSeats` задаётся при создании события и не изменяется.
- `AvailableSeats` уменьшается на 1 при каждом успешном бронировании (метод `TryReserveSeats`) и увеличивается при отмене брони (метод `ReleaseSeats`).

## Фоновая обработка бронирований

В приложении работает фоновая служба `BookingResolverService`, которая:

- Каждые 3 секунды опрашивает хранилище на наличие бронирований со статусом `Pending`.
- Для каждого такого бронирования имитирует обработку (задержка 2 секунды), после чего переводит его в статус `Confirmed` и устанавливает поле `ProcessedAt` в текущее время.
- Обновляет бронирование в хранилище.

Служба реализована как `BackgroundService` и запускается автоматически при старте приложения.

## Примитивы синхронизации

Для защиты от конкурентных проблем использованы следующие механизмы:

1. **SemaphoreSlim в BookingService** – гарантирует, что только один поток одновременно выполняет критическую секцию в `CreateBookingAsync` (проверка доступности мест, уменьшение `AvailableSeats`, сохранение брони). Это предотвращает овербукинг (продажу одних и тех же мест нескольким пользователям).

2. **SemaphoreSlim в BookingResolverService** – защищает обновление хранилища бронирований при параллельной обработке нескольких `Pending`‑бронирований. Семафор ограничивает одновременный доступ к операциям `Confirm()`/`Reject()` и обновлению хранилища.

Использование `SemaphoreSlim` вместо `lock` обусловлено асинхронной природой методов (`await` внутри критической секции).

## Пример сценария использования

1. **Создание события**  
   ```http
   POST /api/events
   Content-Type: application/json

   {
     "id": "12345678-1234-1234-1234-123456789abc",
     "title": "Linkin Park",
     "description": "Концерт Linkin Park в Яндекс Музыке",
     "startAt": "2026-03-22T10:30:00",
     "endAt": "2026-03-22T12:00:00",
     "totalSeats": 10
   }
   ```
   Ответ содержит созданное событие с присвоенным `Id`.

2. **Получение списка событий с фильтрацией**  
   ```http
   GET /api/events?title=Linkin&from=2026-03-21&page=1&pageSize=8
   ```

3. **Создание бронирования**  
   ```http
   POST /api/events/{eventId}/book
   ```
   Ответ:
   - Код `202 Accepted`
   - Заголовок `Location` содержит URL для отслеживания статуса бронирования (например, `/bookings/12345678-1234-1234-1234-123456789abc`)
   - Тело ответа содержит данные созданного бронирования со статусом `Pending`.

4. **Отслеживание статуса бронирования**  
   ```http
   GET /api/bookings/{bookingId}
   ```
   Пока бронирование обрабатывается, статус будет `Pending`. Через несколько секунд повторный запрос вернёт статус `Confirmed` и заполненное поле `ProcessedAt`.

5. **Удаление события**
    ```http
    DELETE /api/events/{eventId}
    ```

## Пример сценария с овербукингом

Допустим, событие имеет `TotalSeats = 5`. Одновременно 20 пользователей отправляют запросы на бронирование. Без синхронизации могло бы создаться 20 броней, хотя мест всего 5. Благодаря `SemaphoreSlim` в `BookingService`:

- Запросы обрабатываются последовательно внутри критической секции.
- Первые 5 запросов успешно резервируют места (`AvailableSeats` уменьшается с 5 до 0).
- Последующие 15 запросов получают ответ `409 Conflict` (исключение `NoAvailableSeatsException`).
- В результате ровно 5 броней получают статус `Pending`, а затем `Confirmed` фоновой службой.

Это гарантирует, что количество проданных билетов никогда не превысит `TotalSeats`.

## Запуск тестов

В проекте присутствует набор Unit тестов, расположенных в папке `Yandex_ASPNET_Ticket_Service.UnitTests`.

Для запуска тестов выполните команду:

```bash
cd .\Yandex_ASPNET_Ticket_Service.UnitTests\
dotnet test
```

Тесты покрывают основные сценарии работы контроллеров, сервисов и хранилищ.

### Использование InMemory-провайдера

В целях изоляции и скорости выполнения тесты используют **InMemory-провайдер Entity Framework Core**. Это позволяет запускать тесты без зависимости от реальной базы данных PostgreSQL. Каждый тестовый класс создаёт свою изолированную in-memory базу данных, которая автоматически очищается после завершения теста.

### Интеграционные тесты

Помимо Unit тестов, проект содержит интеграционные тесты, расположенные в папке `Yandex_ASPNET_Ticket_Service.IntegrationTests`. Эти тесты проверяют взаимодействие с реальной базой данных PostgreSQL, запускаемой в Docker-контейнере с помощью библиотеки Testcontainers.

**Требования для запуска интеграционных тестов:**
- Установленный и запущенный Docker Desktop (или Docker Engine) на вашей машине.
- Доступность демона Docker для создания контейнеров.

**Запуск интеграционных тестов:**
```bash
cd .\Yandex_ASPNET_Ticket_Service.IntegrationTests\
dotnet test
```

Интеграционные тесты автоматически разворачивают временный контейнер PostgreSQL, применяют миграции (через `EnsureCreated`) и выполняют тесты, после чего контейнер удаляется. Это позволяет проверить корректность работы репозиториев и сервисов в условиях, максимально приближенных к production.

> **Важно:** Убедитесь, что Docker запущен перед выполнением интеграционных тестов, иначе тесты завершатся с ошибкой.

## Формат ошибок

API использует стандартный формат ошибок [Problem Details (RFC 7807)](https://datatracker.ietf.org/doc/html/rfc7807). В случае ошибки клиент получит ответ с соответствующим HTTP‑кодом и JSON‑объектом, содержащим поля `type`, `title`, `status`, `detail` и, при необходимости, `instance`.

Пример ответа с ошибкой 404:

```json
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.4",
  "title": "Not Found",
  "status": 404,
  "detail": "Event with id [12345678-1234-1234-1234-123456789abc] not found",
  "instance": "/events/12345678-1234-1234-1234-123456789abc"
}
